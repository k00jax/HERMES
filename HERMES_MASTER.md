## 1. SYSTEM IDENTITY

HERMES is a distributed edge cognition platform composed of:

* Tier 1: Sensor and Interface MCUs
* Tier 2: Edge Compute Node
* Tier 3: AI and Memory Layer

It is designed for:

* Offline-first operation
* Context logging
* Environmental awareness
* Human presence sensing
* Field-ready reliability

It is not a cloud-dependent device.
It must degrade gracefully without AI or network.

## 2. CURRENT HARDWARE ARCHITECTURE

### Tier 1 – Sensor and Interface Layer

**MCU 1: nRF52840 (Primary Interface Controller)**
Responsibilities:

* I2C sensor bus host
* OLED display control
* Button input processing
* Buzzer control
* LED expansion control via shift register
* UART link to ESP32

Connected devices:

* SHT (temperature + humidity)
* SGP (ECO2 / air quality)
* Dual OLED displays (shared I2C bus)
* Passive buzzer (D0)
* Three-button navigation:

  * D1 = Previous
  * D2 = Select
  * D3 = Next
* 74HC595 shift register for:

  * Status LEDs
  * RGB LED control
* RGB LED
* Additional discrete LEDs via 74HC595

All I2C devices share SDA/SCL bus with local decoupling capacitors.

---

**MCU 2: ESP32-S3 (Radar + Telemetry Controller)**
Responsibilities:

* LD2410B-P radar UART integration
* Radar frame parsing
* Telemetry emission
* Serial1 link to Odroid

Connected devices:

* LD2410B-P radar sensor via UART

  * VCC powered from 5V pin
  * TX/RX cross-wired
  * No OUT pin used
* UART1 output to Odroid

Radar telemetry emits:

```
RADAR,alive=1,target=...,move_cm=...,stat_cm=...,detect_cm=...,move_en=...,stat_en=...,frame_ts_ms=...,ts=...
```

Emission:

* Throttled to ≤10Hz
* Immediate on state change
* Alive timeout detection after 1s

Radar is not intended to be always-on in future revisions.

---

### Tier 2 – Edge Compute (Odroid M1S)

Responsibilities:

* Serial ingestion from:

  * nRF
  * ESP32
* SQLite database storage:

  * radar
  * air/env
  * events
  * reports
  * calibration
  * settings
* FastAPI dashboard service
* Report generation
* Calibration workflows
* History queries and CSV export

## 3. DATA FLOW

Radar path:
LD2410 → ESP32 → Serial1 → Odroid → SQLite → Dashboard → Reports

Environmental path:
Sensors → nRF → USB Serial → Odroid → SQLite → Dashboard → Reports

Events:
Generated by:

* Calibration completion
* System alerts
* Radar state anomalies
  Stored in events table.

### Home

* Live charts
* Chart slot persistence (A–D)
* RSSI signal border coloring
* Quick access to Field Mode

---

### Field Mode (/field)

Minimal interface for field use:

* Large presence state badge
* Detect distance readout
* ECO2 / Temp / Humidity / RSSI tiles
* No heavy charts

Designed for readability at distance.

---

### Analytics

Provides:

* Presence minutes by hour (24h)
* ECO2 present vs absent comparison
* Event frequency summary (7d)

All queries time-bounded.

---

### Reports

Selectable presets:

* 24h
* 3d
* 1w
* 1m
* Custom

Generates downloadable HTML report containing:

* Presence summary
* Motion and still signature minutes
* ECO2 summary
* Event breakdown
* Calibration references

Reports stored and listed (last 10).

Range hard capped at 31 days.

---

### Calibration

Implements:

Empty-room calibration workflow:

* 60s capture window
* Progress bar
* Sample counting
* False presence count
* Baseline detect distance
* Noise estimate (MAD or stddev fallback)

On completion:

* Calibration row stored
* Event logged
* Passive buzzer chime triggered

---

### History

Supports:

* Kind filter
* Range presets
* Custom time window
* Text search
* CSV export
* Hard cap on limits

---

### Settings

Persisted fields:

* field_mode_start

## 5. POWER MODEL

Current state:

* 3.3V rail powers MCUs and sensors
* 5V rail powers LD2410 only
* Shared ground
* Decoupling:

  * 100µF electrolytic across rails
  * 0.1µF ceramics near modules

Radar is not architected as permanent always-on load.

Future requirement:

* Idle and peak draw modeling required before wearable enclosure.
